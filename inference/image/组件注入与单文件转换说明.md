# 组件自动注入与单文件补件说明（现状）

本文描述当前图片推理侧的组件装配规则，目标是让后续维护者快速理解：

- Diffusion 主链路与 `POSE` 的 diffusers 支线，如何自动发现 nunchaku 组件
- `low_vram` / `fast_mode` 如何影响“组件替换”“CPU offload”“运行期 fast cache”
- `from_single_file` 场景的基础 `config` 注入与标准组件补件
- `zimage + from_single_file` 场景下单文件权重转换器的触发条件与行为

## 总体调用链（图片推理）

- `image/inferrer.py`：根据 `job_type` 选择 handler，扩散类任务进入 `DiffusionHandler`
- `image/handlers/diffusion_handler.py`：调用 `PipelineLifecycle.create_pipeline()` 获取 pipe 与基础推理参数
- `image/runtime/pipeline_lifecycle.py`：内部通过 `PipelineService.acquire()` 进行 pipeline 装配，并维护缓存签名
- `image/runtime/pipeline_service.py`：
  - `PipelineDetector.get_pipeline()` 选择 pipeline class
  - `ModelLocator.locate()` 决定 `from_pretrained` / `from_single_file`
  - `PipelineDetector.build_pipeline_params()` 构建创建 pipeline 的 kwargs
- `common/pipeline_detector.py`：不再维护一堆 `_inject_*` 分支，统一调用注入器拿 overrides
- `common/pipeline_component_injector.py`：**自动组件发现与单文件补件中心**
- `image/handlers/pose_backends.py`：`POSE` 的 diffusers 分支也复用了同一套自动发现逻辑

## 当前结论

- 对 `DiffusionHandler` 和 `POSE` 的 diffusers 支线，**组件装配不再依赖** `model_cfg.transformer / unet / text_encoder / vae` 等字段
- `model_cfg` 在这条链路里保留下来的主要用途是：
  - 非 diffusers 的特殊分支，例如 `anima`
  - 少量仍未迁移的业务字段，例如 `scheduler_loader` 中的 `prediction_type`
  - `zimage` 的 controlnet 路径覆盖（仍在 `pose_assets.py` 中保留）
- nunchaku 组件是否启用，统一由运行模式决定：
  - `low_vram=True` 或 `fast_mode=True`：启用自动发现与替换
  - `low_vram=True`：额外启用 CPU offload
  - `fast_mode=True`：额外执行 `apply_fast_mode_cache()`

## family 能力表

- `common/Constant.py`：`FAMILY_ALLOWED_KEYS`

这张表现在表示“后端支持自动装配/补件哪些标准槽位”，不是“允许前端通过 `model_cfg` 传哪些组件 key”。

当前主要 family 的能力大致如下：

- `sdxl`：`unet` / `nunchaku_unet` / `text_encoder` / `text_encoder_2` / `vae`
- `flux` / `flux_kontext`：`transformer` / `nunchaku_transformer` / `text_encoder_2` / `nunchaku_text_encoder_2` / `vae`
- `flux2` / `flux2_klein`：`transformer` / `nunchaku_transformer` / `text_encoder` / `nunchaku_text_encoder` / `vae`
- `qwen` / `qwen.edit`：`transformer` / `nunchaku_transformer` / `text_encoder` / `nunchaku_text_encoder` / `vae`
- `zimage` / `chroma`：按各自 family 做标准组件补件，其中 `chroma` 也支持 nunchaku text encoder

## 自动发现规则（nunchaku）

自动发现逻辑集中在：

- `common/pipeline_component_injector.py`

### 何时触发

- `low_vram=True` 或 `fast_mode=True`

### 主量化组件（transformer / unet）

在模型目录中寻找：

- `svdq-{precision}_r{32|64|128}-*.safetensors`

其中 `precision` 来自：

```python
from nunchaku.utils import get_precision
precision = get_precision()
```

选择策略：

- 优先当前平台 precision
- 同一 precision 内优先 `r32`
- 否则按较小的 `r` 选择

family 映射：

- `sdxl`：主量化组件映射到 `unet`
- `flux` / `flux_kontext` / `flux2` / `flux2_klein` / `qwen` / `qwen.edit` / `zimage` / `chroma`：主量化组件映射到 `transformer`

### 文本编码器

优先在模型目录寻找：

- `svdq-{precision}-*.safetensors`
- `awq-{precision}-*.safetensors`

若未命中，则按 family 走 fallback：

- `flux2` / `flux2_klein` -> `Qwen3-text-Nunchaku/svdq-int4-Qwen3-text-Nunchaku.safetensors`
- `qwen` -> `Qwen2.5vl-Nunchaku/svdq-int4-Qwen2.5vl-text-Nunchaku.safetensors`
- `qwen.edit` -> `Qwen2.5vl-Nunchaku/svdq-int4-Qwen2.5vl-Nunchaku.safetensors`
- `flux` / `flux_kontext` / `chroma` -> `nunchaku-t5/awq-int4-flux.1-t5xxl.safetensors`

### 运行模式语义

- `low_vram=True`
  - 启用 nunchaku 自动发现与替换
  - 推理阶段启用 CPU offload
- `fast_mode=True`
  - 启用 nunchaku 自动发现与替换
  - 推理阶段执行 `apply_fast_mode_cache()`
- 两者都不是
  - 不做 nunchaku 自动替换
  - 使用全精度默认 pipeline

### 关键日志

当前自动发现会输出这几类日志：

- `auto-detect nunchaku transformer/unet/text encoder hit`
- `auto-detect miss -> fallback`
- `unsupported family/component`
- `candidate found but load failed`

## from_single_file：标准组件补件

该逻辑用于处理“单文件 ckpt 只包含主权重，缺失标准组件”的情况。

策略表集中在：

- `common/Constant.py`：`FAMILY_SINGLEFILE_POLICY`

## from_single_file 的 config 规则

在 `common/pipeline_component_injector.py` 中：

- 若 `model_info.method == "from_single_file"`，会尝试注入：
  - `config = <repo>/inference/config/<family>`（绝对路径）
- 如果该目录不存在：只打印 **一条 warn**，并 **不注入** `config`（不中断流程）

### 标准组件补件顺序

当 `model_info.method == "from_single_file"` 时，对 `required_components` 按顺序补齐：

1. 自动 overrides
   - 例如前面已经自动发现到的 nunchaku `transformer/unet/text_encoder`
2. 单文件自身
   - 通过扫描 `.safetensors` 的 key 前缀判断是否包含某组件
3. 基模兜底
   - 从 `{models_dir}/{base_path}` 加载缺失的标准组件

### 必选组件校验

若补齐结束后仍缺任何 `required_components`，会直接抛 `ValueError`。

### 来源日志

对每个标准组件都会输出：

- `[single_file_assemble] <family> <component> <- <source>`

其中 `source` 可能为：

- `single_file`
- `single_file(converter)`
- `base_model`
- 自动发现写入的来源标记

## zimage 单文件权重转换器（TE/VAE/Transformer）

### 背景

`zimage` 单文件权重中：

- `text_encoder` 可能是 `Qwen3Model` 等非 CLIP/T5 的结构
- `vae` 可能不是 diffusers 单文件 loader 能自动灌权重的格式

因此仅依赖 `ZImagePipeline.from_single_file()` 往往会：

- 只加载 transformer 主权重
- TE/VAE 被忽略或报错

### 实现

新模块：

- `common/single_file_component_converter.py`

提供按需构建 overrides 的入口：

- `build_zimage_overrides_for_single_file(...) -> {text_encoder?, vae?, transformer?}`

主要逻辑（参考 `test/single_file_zimage.py`）：

- **text_encoder(Qwen3)**：
  - 从 safetensors 按前缀抽取权重
  - 用 `config_dir/text_encoder` 的 config 构建结构
  - 多组候选规则重映射 key，选命中最多的一组
  - `load_state_dict(strict=False)` 并记录 missing/unexpected
- **vae(AutoencoderKL)**：
  - 先尝试直接 strip 前缀匹配
  - 匹配不足则调用 `diffusers.loaders.single_file_utils.convert_ldm_vae_checkpoint` 转换后加载
- **transformer(ZImageTransformer2DModel)**：
  - 从 safetensors 排除 TE/VAE 前缀，只喂 transformer 相关权重
  - 调用 `ZImageTransformer2DModel.from_single_file(tensor_dict, config=..., subfolder="transformer")`

### 触发点

在 `common/pipeline_component_injector.py` 中：

- 仅当 `zimage + from_single_file` 且需要补缺组件时触发
- 若 `inference/config/zimage` 不存在：warn 并跳过转换器
- 若转换器内部报错：warn 并跳过（不中断，让 diffusers 自己尝试）

## pipeline 缓存签名更新

为了避免“装配结果变化但缓存 key 不变导致误复用”，`image/runtime/pipeline_lifecycle.py` 现在主要依赖：

- `pipeline_params["__component_sig"]`
- `from_single_file` 相关的 `config / original_config / cache_dir / use_safetensors`
- `low_vram`
